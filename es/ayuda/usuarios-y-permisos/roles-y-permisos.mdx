Los roles agrupan permisos. Un permiso autoriza una capacidad concreta, como leer pedidos de venta, crear recepciones o trabajar una orden de fabricación.

Un usuario puede tener varios roles. Bold combina sus permisos para decidir qué módulos ve y qué acciones puede realizar.

## Usuario, empleado, rol y permiso

| Concepto | Qué representa | Ejemplo |
| --- | --- | --- |
| Usuario | La cuenta de acceso a Bold, normalmente identificada por email. | `ana@empresa.com` |
| Empleado | La persona operativa para turnos, tiempos, asignaciones y trabajo. | Ana García, operaria de línea 2. |
| Rol | Un paquete reutilizable de permisos. | Responsable de almacén. |
| Permiso | Una acción concreta dentro de un módulo. | `Warehouse.Receipts.Write` |

<Info>
  Usuario y empleado no son lo mismo. Vincula ambos cuando la persona debe registrar trabajo operativo, tiempos o acciones en planta.
</Info>

## Qué contiene un rol

Cada rol tiene:

- **Nombre**, visible para administradores.
- **Descripción**, útil para explicar cuándo usarlo.
- **Clave**, usada internamente por la identidad.
- Permisos asignados.
- Indicador de rol de sistema.

Los roles cuya clave empieza por `App` son roles de sistema. Por ejemplo, las acciones de administración de usuarios y roles requieren `App.Admin`.

## Patrón de permisos

Los permisos siguen una estructura legible:

```text
Modulo.Recurso.Accion
```

Ejemplos:

| Permiso | Qué permite |
| --- | --- |
| `Items.Skus.Read` | Leer artículos y SKUs. |
| `Warehouse.Receipts.Write` | Crear o modificar recepciones. |
| `Planning.SalesOrders.Write` | Crear o modificar pedidos de venta. |
| `Production.Orders.Work` | Ejecutar trabajo en órdenes de fabricación. |
| `Production.Recipes.Activate` | Activar versiones de receta. |

Las acciones más comunes son `Read` y `Write`, pero hay permisos específicos para tareas críticas, como `Work`, `Sequence`, `Activate`, `Presence`, `Upload` o `Delete`.

## Crear un rol útil

<Steps>
  <Step title="Define la responsabilidad">
    Describe qué debe hacer la persona en lenguaje operativo: recibir material, planificar compras, ejecutar órdenes o mantener activos.
  </Step>
  <Step title="Añade acceso común">
    Asegúrate de que el rol permite entrar y usar el modo de aplicación que necesita.
  </Step>
  <Step title="Añade lectura primero">
    Da visibilidad a los módulos que la persona necesita consultar.
  </Step>
  <Step title="Añade escritura solo donde actúa">
    Limita cambios a las áreas donde la persona tiene responsabilidad directa.
  </Step>
  <Step title="Separa acciones críticas">
    Mantén activar recetas, secuenciar órdenes, gestionar roles, borrar archivos o administrar credenciales en roles controlados.
  </Step>
  <Step title="Prueba con un usuario real">
    Comprueba navegación, botones visibles y errores de permisos antes de reutilizar el rol.
  </Step>
</Steps>

## Ejemplos de diseño

| Responsabilidad | Permisos habituales |
| --- | --- |
| Operario de planta | Acceso común, lectura de producción y `Production.Orders.Work`. |
| Responsable de almacén | Lectura y escritura de ubicaciones, stock, recepciones, envíos, lotes e inventarios. |
| Planificador | Lectura y escritura de planificación, lectura de catálogo, compras y producción según proceso. |
| Supervisor de mantenimiento | Lectura y escritura de activos, causas, acciones, partes y preventivos. |
| Consulta directiva | Lectura de módulos relevantes y reportes, sin escritura operativa. |
| Administrador | `App.Admin` más permisos funcionales que necesite para revisar el sistema. |

<Warning>
  No uses un rol amplio para resolver un bloqueo puntual. Ajusta el permiso concreto o crea un rol temporal revisable.
</Warning>

## Buenas prácticas

- Crea roles por responsabilidad, no por nombre de persona.
- Mantén roles pequeños y combinables.
- Separa lectura de escritura cuando varias personas consultan, pero pocas modifican.
- Revisa roles cuando una persona cambia de puesto.
- Retira accesos antiguos antes de añadir responsabilidades nuevas.
- Documenta en la descripción para qué sirve cada rol.

## Problemas habituales

| Problema | Qué revisar |
| --- | --- |
| El usuario no ve un módulo | Falta acceso común o permiso `Read` del módulo. |
| El usuario ve el módulo, pero no puede guardar | Falta permiso `Write` o una acción específica. |
| No puede trabajar en planta | Revisa `Production.Orders.Work` y la vinculación con empleado. |
| No puede administrar usuarios o roles | Las acciones administrativas requieren `App.Admin`. |
| Tiene demasiadas opciones visibles | Revisa roles acumulados y permisos heredados por varios roles. |
